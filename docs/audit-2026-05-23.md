## Findings

### HIGH — Staff/Manager permissions are not enforced on most server routes (latent)
The permission model (`src/lib/permissions.ts`) and the UI to assign permissions
are built and working, and **data scoping is enforced** on the clients list and
the schedule (page-level queries via `scopeForMember`). But:

- **`TRAINER_NAV` is static** (`src/components/shared/app-shell.tsx`) — every
  trainer sees every nav item regardless of role.
- **Management API routes don't check permissions.** Only 4 of 134 API route
  files use `getTrainerContext`/`requirePermission`. `/api/packages`,
  `/api/products`, `/api/class-runs`, `/api/enquiries`, `/api/achievements`,
  `/api/library/*`, `/api/templates/*`, `/api/embed-forms`, `/api/session-forms`
  scope only by `trainerId` (= the company), so a STAFF member could mutate them.
- **`getClientAccess` (`src/lib/trainer-access.ts`) keys off owning a
  `TrainerProfile`.** A staff member has a membership, not a profile, so it
  returns `null` → client *detail / edit / packages / dogs / field-values*
  routes **404 for staff**. This is *fail-closed* (secure) but means a staff
  member can't open the clients they're assigned to. Needs to resolve via
  membership instead.

**Why it's not an emergency:** the only trainer in the customer's business is the
OWNER (full access by design). OWNER bypasses every gate, so nothing about their
experience is affected. The gap only matters once a MANAGER/STAFF is invited.

**Fix (a designed pass, ideally with Karl on the product calls):**
1. Update `getClientAccess` to resolve the company via membership (so members
   get scoped access), and scope to `assignedMembershipId` when the member lacks
   `clients.viewAll`.
2. Add `requirePermission(...)` guards to the mutating handlers of the management
   routes, mapping each feature area to its permission. OWNER/MANAGER bypass per
   the role presets, so the customer is unaffected.
3. Gate `TRAINER_NAV` by permission in the trainer layout (UX honesty).
4. Extend the permission catalogue to cover areas it currently omits:
   **achievements, awards, ai-tools** (and decide library-vs-templates split).

### MEDIUM — Migration history doesn't replay from zero
`prisma migrate deploy` against an empty DB fails (`20260428_add_product_requests`
references `products` before it exists). Prod is fine (history applied
incrementally) and the E2E harness side-steps it with `db push`, but a clean
rebuild / new environment can't `migrate deploy` from scratch. Worth squashing
the migration history at some point.

### LOW — Pre-existing lint errors
~9 ESLint errors in `schedule-view.tsx` and `clients/*` (`react/no-unescaped-entities`,
React-Compiler "impure function during render" on `new Date()` inside query
where-clauses). Pre-existing, not build-breaking in Next 16 (ESLint decoupled
from `next build`). Cosmetic; clean up opportunistically.

### Fixed during this audit
- **`vitest` had no config** → its default glob ran the Playwright `*.spec.ts`
  files and errored. Added `vitest.config.ts` scoping to `tests/unit`. `npm test`
  is now clean (49/49).

## Recommendations (priority order)

1. **Before inviting any staff/manager:** do the permission-enforcement pass
   (HIGH finding) — `getClientAccess` via membership + route guards + nav gating
   + catalogue gaps. This is the one thing standing between "works for solo
   owners" and "works for teams".
2. **Grow the E2E suite** (`tests/e2e/`, now isolated + repeatable) to cover the
   core owner flows button-by-button: create/invite client, schedule a session,
   assign a package, group classes, products, enquiries → reply/accept, messages.
   This is the realistic route to "test every feature every button" repeatably.
3. Squash migration history so fresh environments can `migrate deploy`.
4. Opportunistic lint cleanup.
